1. PROJECT OVERVIEW & VISION
================================================================================

Snow Mind is an autonomous yield optimization agent built on Avalanche. Users
deposit assets into a smart account, and an off-chain optimization engine
continuously reallocates capital across whitelisted lending/yield protocols to
maximize risk-adjusted returns — without the user needing to manually manage
positions.

Core thesis: Most "AI yield agents" in DeFi use traditional mathematical
optimization (MILP solvers, mean-variance optimization, Monte Carlo simulations)
for their core allocation logic — not LLMs. We follow the same proven approach
for the optimizer, while using the Kite AI ecosystem and x402 protocol for
auxiliary data services (sentiment, risk signals, market intelligence).

Target chain: Avalanche C-Chain (primary), with architecture designed for
future multi-chain expansion.

Target user: DeFi-native users with $5K-$500K in stablecoins or blue-chip
assets who want passive yield optimization without active management.

Data flow:
1. User creates a ZeroDev smart account via our frontend
2. User deposits assets (USDC, USDT, AVAX, etc.) into their smart account
3. User grants a session key to the Snow Mind optimizer backend
4. Optimizer periodically fetches current yield rates across all protocols
5. MILP solver computes optimal allocation given constraints
6. Optimizer generates + signs rebalance txns using the session key
7. Transactions are submitted via an ERC-4337 bundler
8. User can revoke session key or withdraw at any time

3. SMART ACCOUNT ARCHITECTURE (ZERDEV KERNEL)
================================================================================

Why ZeroDev Kernel:
- Open source smart account framework (ERC-4337 compatible)
- Used by Giza (confirmed via reverse engineering) and Glider (a16z-backed)
- Modular plugin/validator system for adding custom logic
- Session key support built-in via SessionKeyValidator
- Battle-tested: Kernel v0.3.3 is the version Giza deploys on-chain

Key Components:

3.1 KERNEL SMART ACCOUNT
- Each user gets a Kernel smart account deployed on Avalanche
- This is NOT a multisig — it's a single-owner smart account with programmable
  validation logic
- The user's EOA (MetaMask, etc.) is the root owner/signer
- Smart account holds all user funds (fully non-custodial)

Implementation:
- Use @zerodev/sdk to deploy Kernel accounts
- Kernel v0.3.3 contracts (verify Avalanche deployment — if not deployed,
  we deploy ourselves or use latest available version)
- Entry point: standard ERC-4337 EntryPoint contract on Avalanche

3.2 SESSION KEY VALIDATOR
- The session key is the core mechanism that lets our optimizer execute
  transactions on behalf of the user WITHOUT holding their private key
- Session keys are scoped — they can only call specific contracts, specific
  functions, with specific parameter constraints

Session Key Configuration for Snow Mind:
  {
    // Allowed target contracts (whitelist only)
    allowedContracts: [
      BENQI_LENDING_POOL,
      AAVE_V3_POOL,
      EULER_V2_VAULT,
      FLUID_VAULT
    ],

    // Allowed functions
    allowedFunctions: [
      "supply(address,uint256,address,uint16)",
      "withdraw(address,uint256,address)",
      "deposit(uint256,address)",
      "redeem(uint256,address,address)"
    ],

    // Constraints
    maxTransactionValue: USER_DEFINED_CAP,
    expirationTimestamp: NOW + 30_DAYS,  // renewable
    rateLimit: MAX_N_TRANSACTIONS_PER_DAY
  }

Critical security properties:
- Session key CANNOT transfer funds to arbitrary addresses
- Session key CANNOT call unapproved contracts
- Session key has an expiration — user must re-authorize periodically
- User can revoke the session key at any time via their root key

3.3 ERC-4337 BUNDLER
- We need a bundler service to submit UserOperations to the EntryPoint
- Options: Pimlico, Stackup, Alchemy, or self-hosted (Infinitism reference)
- For MVP: use Pimlico or Stackup's hosted bundler on Avalanche
- The bundler pays gas (we can sponsor via a Paymaster, or user pre-funds gas)

Decision needed: Do we want to sponsor gas for users via a Paymaster contract,
or require users to hold AVAX for gas? (Recommendation: sponsor gas for MVP
to reduce friction — cost is minimal for hackathon scale)

3.4 ACCOUNT DEPLOYMENT FLOW
1. User connects wallet (EOA) on frontend
2. Frontend calls ZeroDev SDK to compute counterfactual smart account address
3. Smart account is deployed on first transaction (lazy deployment)
4. User signs a session key authorization (EIP-712 typed signature)
5. Session key is stored by our backend (encrypted)
6. Optimizer is now authorized to act within session key scope

4. MILP YIELD OPTIMIZER DESIGN
================================================================================

Why MILP (Mixed-Integer Linear Programming):
- This is what the top projects actually use (Giza uses MILP, confirmed
  from their architecture). Almanak uses Monte Carlo. Theoriq uses
  mean-variance. None of them use LLMs for core allocation.
- MILP is deterministic, auditable, and fast for portfolio allocation problems
- Can express complex constraints (min/max per protocol, risk limits, etc.)
- Well-supported by open-source solvers

4.1 THE OPTIMIZATION PROBLEM

Objective function: Maximize risk-adjusted yield across N protocols

  maximize: Σ(allocation_i × yield_i) - λ × risk_penalty

Subject to constraints:
  - Σ(allocation_i) = total_portfolio_value          (fully allocated)
  - allocation_i >= 0                                 (no negative positions)
  - allocation_i <= max_allocation_per_protocol       (diversification cap)
  - allocation_i <= protocol_i_tvl × max_share        (don't exceed X% of protocol TVL)
  - risk_score_i <= max_acceptable_risk               (risk threshold)

Variables:
  - allocation_i: amount deposited in protocol i (continuous variable)
  - yield_i: current APY/APR of protocol i (input parameter, fetched on-chain)
  - risk_score_i: protocol risk rating (input parameter, from risk model)
  - λ (lambda): risk aversion parameter (user-configurable or default)

4.2 INPUT DATA PIPELINE

For each integrated protocol, fetch:
  - Current supply APY (on-chain query or subgraph)
  - Total TVL / utilization rate
  - Available liquidity for deposits
  - Any reward token APY (AVAX rewards, protocol tokens, etc.)

Data sources:
  - Direct smart contract reads (preferred — most accurate, real-time)
  - Subgraphs (The Graph) for historical data
  - DefiLlama API for cross-protocol comparison / sanity checks
  - (Post-MVP) Kite AI data services via x402 for sentiment/risk signals

Refresh frequency:
  - MVP: Every 30 minutes (cron job)
  - Post-MVP: Event-driven (monitor for significant rate changes)

4.3 RISK MODEL (SIMPLIFIED FOR MVP)

For MVP, use a simple weighted risk score per protocol:

  risk_score = w1 × smart_contract_risk
             + w2 × liquidity_risk
             + w3 × concentration_risk

Where:
  - smart_contract_risk: static rating (1-10) based on audit status, age,
    TVL history. Manually assigned for each protocol initially.
  - liquidity_risk: dynamic — based on current utilization rate and
    available liquidity. Higher utilization = higher risk.
  - concentration_risk: dynamic — if Snow Mind users collectively represent
    >X% of a pool, increase risk score.

Post-MVP: Replace static ratings with dynamic scoring from on-chain
analytics, integrate insurance protocol data, add volatility metrics.

4.4 SOLVER IMPLEMENTATION

Recommended solver: Google OR-Tools (open source) or PuLP (Python)
- Both support MILP natively
- OR-Tools is faster for larger problem instances
- PuLP is simpler to set up and sufficient for MVP (4 protocols)

Backend stack recommendation:
- Python backend (FastAPI or Flask)
- PuLP or OR-Tools for MILP solving
- Web3.py for on-chain data fetching
- APScheduler or Celery for periodic optimization runs

Solve time: With 4 protocols and simple constraints, solve time will be
<1 second. This is trivial.

4.5 REBALANCE LOGIC

The optimizer doesn't rebalance on every run. It follows a threshold model:

  IF (proposed_allocation - current_allocation) > rebalance_threshold:
      generate_rebalance_transactions()

Rebalance threshold: configurable, default 5% of portfolio per protocol.
This prevents excessive gas costs from micro-adjustments.

Transaction generation:
1. Calculate delta between current and proposed allocation per protocol
2. Generate withdraw transactions for protocols being reduced
3. Generate supply transactions for protocols being increased
4. Order: withdrawals first, then deposits (ensure funds are available)
5. Bundle into a batch UserOperation via the smart account
6. Sign with session key
7. Submit via bundler

Gas optimization: Kernel smart accounts support batch execution — multiple
calls in a single UserOperation. This saves gas vs. individual transactions.

5. PROTOCOL INTEGRATIONS
================================================================================

MVP integrations (Avalanche C-Chain):

5.1 BENQI (benqi.fi)
- Type: Lending/borrowing (Compound fork)
- Contracts: qiTokens (supply token model)
- Key functions:
    mint(uint256 amount) — supply assets, receive qiTokens
    redeem(uint256 qiTokenAmount) — withdraw assets
    redeemUnderlying(uint256 underlyingAmount) — withdraw exact amount
- APY source: On-chain via exchangeRateStored() + supply rate per block
- Reward tokens: QI token rewards (check if still active)
- Supported assets: USDC, USDT, AVAX, BTC.b, WETH
- Risk rating: Medium-low (established, audited, high TVL on Avalanche)
- Notes: Benqi is the dominant lending protocol on Avalanche — this is our
  primary integration and should be rock-solid

5.2 AAVE V3 (aave.com)
- Type: Lending/borrowing
- Contracts: Aave V3 Pool contract on Avalanche
- Key functions:
    supply(address asset, uint256 amount, address onBehalfOf, uint16 refCode)
    withdraw(address asset, uint256 amount, address to)
- APY source: Pool.getReserveData() returns currentLiquidityRate
- Reward tokens: Check Avalanche incentives controller
- Supported assets: USDC, USDT, DAI, AVAX, WETH, BTC.b
- Risk rating: Low (battle-tested across multiple chains, top-tier audits)
- Notes: Aave V3 has efficiency mode (eMode) — could be leveraged later
  for correlated asset optimization

5.3 EULER V2 (euler.finance)
- Type: Modular lending (vault-based architecture)
- Contracts: ERC-4626 vault interface
- Key functions:
    deposit(uint256 assets, address receiver) — deposit into vault
    redeem(uint256 shares, address receiver, address owner) — withdraw
    withdraw(uint256 assets, address receiver, address owner) — withdraw exact
- APY source: Vault-specific, query via totalAssets()/totalSupply() changes
- Risk rating: Medium (v2 is newer, modular design adds complexity)
- Notes: Euler v2 is vault-based (ERC-4626), which is a cleaner interface
  than Compound-fork models. Verify Avalanche deployment status.

  ** IMPORTANT: Confirm Euler v2 is live on Avalanche before committing
  engineering time. If not deployed, deprioritize for MVP. **

5.4 FLUID (fluid.instadapp.io)
- Type: Unified lending/borrowing with smart collateral
- Contracts: Fluid lending vaults
- Key functions: Varies by vault type — investigate specific ABI
- APY source: Vault-specific rate queries
- Risk rating: Medium (newer protocol, backed by Instadapp team)
- Notes: Fluid is interesting for post-MVP because of "smart debt" and
  "smart collateral" features. For MVP, only use simple supply positions.

  ** IMPORTANT: Same as Euler — verify Avalanche deployment. If only on
  mainnet Ethereum, deprioritize. **

5.5 INTEGRATION PRIORITY ORDER (MVP)
1. Benqi — MUST HAVE (primary Avalanche lending protocol)
2. Aave V3 — MUST HAVE (proven, high TVL on Avalanche)
3. Euler v2 — NICE TO HAVE (if deployed on Avalanche)
4. Fluid — NICE TO HAVE (if deployed on Avalanche)

Minimum viable product requires Benqi + Aave V3. Two protocols is enough
to demonstrate autonomous rebalancing.

5.6 APPROVAL FLOW
Before the optimizer can supply to any protocol, the smart account must
approve token spending. This happens once per token per protocol:

  smart_account.execute(
    token.approve(PROTOCOL_ADDRESS, MAX_UINT256)
  )

This approval transaction uses the session key and should be done during
initial setup or first rebalance.

6. SECURITY MODEL & SESSION KEYS
================================================================================

Security is THE critical concern for an autonomous agent that controls
user funds. Our architecture must ensure:

6.1 NON-CUSTODIAL GUARANTEE
- User funds NEVER leave their smart account
- Our backend NEVER has access to the user's root private key
- Our backend only holds a scoped session key
- The session key cannot move funds to any address outside the whitelisted
  protocol contracts

6.2 SESSION KEY SCOPING (DEFENSE IN DEPTH)

Layer 1 — Contract whitelist:
  Session key can ONLY interact with approved protocol contracts
  (Benqi, Aave, Euler, Fluid lending pools)

Layer 2 — Function whitelist:
  Session key can ONLY call supply/withdraw functions
  Cannot call borrow, liquidate, transfer, or any other function

Layer 3 — Parameter constraints:
  Maximum transaction value per call
  Maximum number of transactions per time period
  Cannot supply to addresses not in the approved asset list

Layer 4 — Temporal constraints:
  Session key expires after a set period (e.g., 30 days)
  Must be re-authorized by user

Layer 5 — User override:
  User can revoke session key at ANY time from the frontend
  Revocation is immediate (on-chain state change)
  User can always directly interact with their smart account using root key

6.3 BACKEND KEY MANAGEMENT
- Session key private key stored encrypted at rest (AES-256)
- Decrypt only in memory at time of transaction signing
- Backend server should run in a secure environment
- For MVP: encrypted env variables are acceptable
- Post-MVP: consider TEE (Trusted Execution Environment) like Almanak uses,
  or AWS KMS / HashiCorp Vault

6.4 RISK SCENARIO ANALYSIS

Scenario: Our backend server is compromised
  Impact: Attacker gains session key
  Mitigation: Session key can ONLY supply/withdraw to whitelisted protocols.
  Attacker cannot steal funds (no transfer permission). Worst case: attacker
  makes suboptimal allocations. User can revoke and withdraw immediately.

Scenario: A whitelisted protocol is exploited
  Impact: Funds in that protocol are at risk (same as any DeFi user)
  Mitigation: Diversification cap (max X% per protocol) limits exposure.
  Post-MVP: automated emergency withdrawal on exploit detection.

Scenario: Smart contract bug in Kernel
  Impact: Potentially severe
  Mitigation: ZeroDev Kernel is audited and battle-tested. We use a known
  stable version (v0.3.3). We do not write custom validator plugins for MVP.

6.5 EMERGENCY PROCEDURES
- Frontend should have a "PANIC BUTTON" that:
  1. Revokes all session keys
  2. Withdraws all positions to the smart account
  3. User can then transfer to their EOA
- This should work even if our backend is completely down

7. KITE AI / x402 DATA SERVICE INTEGRATION
================================================================================

This is our Avalanche ecosystem differentiation. Kite AI is Avalanche's
AI infrastructure layer, and x402 is the HTTP-native payment protocol for
agent-to-agent transactions.

7.1 MVP SCOPE (LIGHT INTEGRATION)
For the hackathon, the Kite AI integration is a value-add, not the core
product. The core product is the MILP optimizer + ZeroDev smart accounts.

MVP Kite integration:
- Register Snow Mind as an agent on Kite's agent registry
- Demonstrate consuming at least one data service via x402
  (e.g., risk scoring, sentiment analysis, or yield prediction)
- Use the data as an input signal to the risk model (Section 4.3)

7.2 x402 PAYMENT FLOW
x402 is a protocol where AI agents pay for API calls via HTTP headers:

  1. Agent makes HTTP request to a data provider
  2. Provider responds with 402 Payment Required + payment details
  3. Agent creates a crypto payment (small amount, USDC on Avalanche)
  4. Agent re-sends request with payment proof in headers
  5. Provider validates payment and returns data

For Snow Mind: Our optimizer backend acts as the paying agent. When it
needs supplementary data (risk score, market sentiment), it calls a
Kite-registered data provider and pays via x402.

This demonstrates the "agent economy" thesis — our optimizer agent is
an autonomous economic actor that consumes and pays for services.

7.3 POST-MVP VISION
- Snow Mind itself becomes a data provider: sell anonymized, aggregated
  yield strategy signals to other agents via x402
- Consume multiple specialized data services for richer risk modeling
- Full participation in Kite's agent-to-agent marketplace

9. COMPETITIVE INTELLIGENCE (FROM GIZA REVERSE-ENGINEERING)
================================================================================

This section documents what we learned from analyzing competitors. Use this
to inform architectural decisions and avoid reinventing the wheel.

9.1 GIZA
- Smart accounts: ZeroDev Kernel v0.3.3 (confirmed on-chain)
- Optimization: MILP solver (not LLMs — despite marketing)
- Data: Yield curve modeling, on-chain rate fetching
- Session keys: Standard ZeroDev session key validator
- Takeaway: WE ARE USING THE SAME CORE ARCHITECTURE. This validates our
  approach. Our differentiator is Avalanche-native + Kite AI integration.

9.2 ALMANAK ($14.25M TVL)
- Smart accounts: Gnosis Safe + Zodiac module system
- Optimization: Monte Carlo simulations + Python strategy scripts
- Security: TEE (Trusted Execution Environments) for strategy execution
- Agent framework: Custom — users write Python strategies
- Takeaway: More complex architecture than we need. Safe + Zodiac is
  heavier than ZeroDev Kernel. TEE is interesting for post-MVP.

9.3 GLIDER ($4M raised, a16z CSX)
- Smart accounts: ZeroDev Kernel (same as us and Giza)
- Optimization: Solver network model
- Natural language: Accepts portfolio intents in plain English
- Takeaway: Closest architectural comp to us. They're on different chains
  though. Their NLP intent layer is a nice UX feature for post-MVP.

9.4 THEORIQ / ALPHAVAULT ($23M TVL)
- Optimization: Mean-variance optimization (Markowitz model)
- Approach: More traditional quant finance
- Takeaway: Simpler math model than MILP. MILP is more flexible.

9.5 SAIL (on Base)
- Type: Auto-rotating yield optimizer
- Approach: Rule-based rotation between highest-yield protocols
- Takeaway: Simpler than MILP but less sophisticated. Good validation
  that "auto yield optimizer" is a real product category.

KEY COMPETITIVE INSIGHT: Nobody is doing this well on Avalanche specifically.
The Avalanche DeFi ecosystem is underserved by autonomous yield tools
compared to Ethereum mainnet, Base, and Arbitrum. This is our opening.
